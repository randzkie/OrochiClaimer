Why the API Returns a Challenge
Ang API ay hindi nagbibigay ng ECDSA proof directly. Instead, gumagamit ng challenge-response flow:

1. Hinihingi ng API na i-prove mo na ikaw ang may-ari ng wallet address.
2. Binibigyan ka ng challenge message na kailangan mong i-sign.
3. Kapag nag-sign ka, pinapadala mo yung signature pabalik sa API.
4. Ang API nag-ve-verify ng signature mo, tapos nag-ge-generate ng operator ECDSA proof para sa iyo.
5. Ginagamit mo yung proof para i-call ang claim() sa contract.
// What the API returns (Step 2) { "challenge": "Orochi Network Airdrop Challenge: 73c47b..." } // What you do (Step 3) // Sign the challenge string with your wallet signature = await signer.signMessage(challenge) // What you send back (Step 4) POST /airdropChallenge { "address": "0xYourAddress", "challenge": "Orochi Network Airdrop Challenge: 73c47b...", "signature": "0xa8bbff...1c" } // What the API returns (operator proof) { "ecdsaProof": "0x...", "amount": 30000000000000000000, "nonce": 0, "timestamp": 1700000000 } // What you submit on-chain (Step 5) airdropContract.claim(ecdsaProof, {amount, nonce, timestamp})
5-Minute Expiry
Ang operator proof ay valid lang sa loob ng 5 minutes. Kung hindi mo na-submit bago mag-expire, kailangan mo ulit buong process: fetch challenge → sign → get proof → claim.
